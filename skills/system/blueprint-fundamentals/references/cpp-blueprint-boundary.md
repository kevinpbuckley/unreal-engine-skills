# C++ and Blueprint boundary — deep reference

Deep dive for [../SKILL.md](../SKILL.md). Covers the C++ specifiers that control what is visible
and callable from Blueprint, the `BlueprintNativeEvent` / `BlueprintImplementableEvent` call
dispatch, `UCLASS` specifiers that govern Blueprint subclassing, and practical patterns for a
clean C++ base + Blueprint-subclass architecture. Grounded in UE 5.7
(`Engine/Source/Runtime/CoreUObject/Public/UObject/ObjectMacros.h`).

See also `blueprint-cpp-integration` for the full specifier reference.

## UCLASS specifiers that govern Blueprint subclassing

| Specifier | Effect |
|---|---|
| `Blueprintable` | This C++ class can be subclassed in Blueprint. Required for designers to create a BP based on your class. |
| `NotBlueprintable` | Prevents Blueprint subclassing. Use on internal base classes you don't want exposed. |
| `BlueprintType` | Instances of this class can be used as Blueprint variable types (i.e. you can declare a variable of this class type in a BP). |

```cpp
UCLASS(Blueprintable, BlueprintType)
class MYGAME_API AWeapon : public AActor
{
    GENERATED_BODY()
    // ...
};
```

A class can be `BlueprintType` without being `Blueprintable` — structs are a common example:
you can have a Blueprint variable of struct type without being able to subclass the struct.

## UPROPERTY specifiers for Blueprint access

| Specifier | What the Blueprint can do |
|---|---|
| `BlueprintReadWrite` | Get and set in any Blueprint graph |
| `BlueprintReadOnly` | Get only; no setter exposed |
| `EditAnywhere` | Editable in actor Details panel (implies instance-editable, not Blueprint-callable) |
| `VisibleAnywhere` | Readable in Details panel, not modifiable |
| `Replicated` | Value replicated across the network (see `networking-and-replication`) |
| `ExposeOnSpawn` | Available as a "spawn parameter" on SpawnActor/SpawnActorDeferred nodes |

```cpp
UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Weapon")
float Damage = 25.f;

UPROPERTY(VisibleAnywhere, BlueprintReadOnly, Category="Weapon")
TObjectPtr<UStaticMeshComponent> WeaponMesh;
```

## UFUNCTION specifiers for Blueprint access

| Specifier | Effect |
|---|---|
| `BlueprintCallable` | Appears as an impure callable node (has exec pins) |
| `BlueprintPure` | Appears as a pure node (no exec pins; return value is evaluated lazily) |
| `BlueprintImplementableEvent` | C++ declares but does not implement; Blueprint provides the body |
| `BlueprintNativeEvent` | C++ provides a default body (`_Implementation`); Blueprint can override |

```cpp
// Callable from Blueprint, C++ provides implementation.
UFUNCTION(BlueprintCallable, Category="Weapon")
void Fire();

// Blueprint must implement this; C++ calls it.
UFUNCTION(BlueprintImplementableEvent, Category="Weapon")
void OnFired();

// C++ provides default; Blueprint can override.
UFUNCTION(BlueprintNativeEvent, Category="Weapon")
float GetDamageMultiplier() const;
// .cpp: float AWeapon::GetDamageMultiplier_Implementation() const { return 1.f; }
```

## Call dispatch for BlueprintNativeEvent and BlueprintImplementableEvent

When C++ calls `GetDamageMultiplier()` on an instance, the engine checks the virtual dispatch
table. The compiler emits a thunk for `BlueprintNativeEvent` functions that:

1. Checks whether the `UBlueprintGeneratedClass` has overridden the function.
2. If yes, invokes the Blueprint bytecode.
3. If no, calls the `_Implementation` C++ body.

For `BlueprintImplementableEvent`, step 3 is a no-op (empty stub). The generated thunk is what
makes the call transparent: C++ just calls the function name; routing is automatic.

Because the thunk goes through the `UFunction` / VM machinery, calling
`BlueprintNativeEvent`/`BlueprintImplementableEvent` functions is slower than a direct virtual
C++ call. For hot paths, prefer a plain virtual function with a `BlueprintCallable` wrapper that
delegates to it, or move the logic fully into C++.

## Calling Blueprint from C++

For `BlueprintImplementableEvent` and `BlueprintNativeEvent` functions declared on the C++ class,
C++ calls them by name and the VM routes to the Blueprint body if one exists:

```cpp
// C++ side — just call the function; VM routes to Blueprint implementation if present.
OnFired();  // Calls Blueprint override if the BP provides one, otherwise the C++ stub.
```

For one-off invocations of Blueprint-only custom events or to call across class boundaries, use
Blueprint interfaces — implement the interface in both C++ and Blueprint and call via:

```cpp
IMyInterface::Execute_MyEvent(TargetObject, /* args */);
```

The `Execute_` thunk is generated by UHT for every `BlueprintImplementableEvent` on an interface.

## TSubclassOf — letting designers choose the Blueprint subclass

A common pattern: C++ defines a `TSubclassOf` property so a Blueprint or designer picks which
BP subclass to spawn:

```cpp
UPROPERTY(EditAnywhere, BlueprintReadWrite, Category="Spawning")
TSubclassOf<AWeapon> WeaponClass;

// In BeginPlay or a function:
if (WeaponClass)
{
    AWeapon* W = GetWorld()->SpawnActor<AWeapon>(WeaponClass, GetActorTransform());
}
```

Spawning `WeaponClass` (rather than the raw C++ class) ensures that Blueprint-authored defaults,
component overrides, and Construction Script logic from the chosen subclass are all applied.

## What belongs in C++ vs Blueprint

| C++ is the right place | Blueprint is the right place |
|---|---|
| Core gameplay logic, state machines | Per-asset configuration (mesh, sound, material) |
| Replication and RPCs | Visual and animation glue |
| Performance-critical loops | Simple event responses |
| APIs used by many systems | Prototyping new game feel |
| Anything needing unit tests | Content-creator tunable values |
| Stable base class hierarchy | Input/UI wiring |

**Binary merge risk:** Blueprint assets are binary and don't text-merge. Keep them thin. Two
people editing the same BP simultaneously will require manual merge or one person loses work.

## Version notes

- `ObjectMacros.h` enumerates `BlueprintImplementableEvent` at line 950, `BlueprintNativeEvent`
  at 955, `BlueprintPure` at 984, `BlueprintCallable` at 987 (UE 5.7).
- The `_Implementation` naming convention and UHT-generated thunks are stable across UE4/UE5.
- `TObjectPtr<T>` replaced raw `T*` for UPROPERTY member declarations in UE5; raw `T*` still
  compiles but won't get the added access tracking. See `memory-and-gc`.
