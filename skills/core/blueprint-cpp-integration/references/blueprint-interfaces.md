# Blueprint interfaces — full reference

Deep dive for [../SKILL.md](../SKILL.md). Covers interface declaration, all three function types
(C++ only, `BlueprintImplementableEvent`, `BlueprintNativeEvent`), the dispatch rules, `Cast<>`
vs. `Execute_` vs. `Implements<>`, and `TScriptInterface` safe storage.
Grounded in UE 5.7:
- `Engine/Source/Runtime/CoreUObject/Public/UObject/Interface.h`
- `Engine/Source/Runtime/CoreUObject/Public/UObject/ObjectMacros.h` (namespace `UI`, lines 925–940)
- Official [Interfaces](https://dev.epicgames.com/documentation/unreal-engine/interfaces-in-unreal-engine) doc.

## Interface declaration (always two classes)

Every Unreal interface requires exactly two class declarations in one header, both needed:

```cpp
// MyInteractable.h
#pragma once
#include "UObject/Interface.h"
#include "MyInteractable.generated.h"

// U-prefixed class: empty; gives the reflection system a UClass to work with
UINTERFACE(MinimalAPI, Blueprintable)
class UMyInteractable : public UInterface { GENERATED_BODY() };

// I-prefixed class: the actual interface — holds all function declarations
class IMyInteractable
{
    GENERATED_BODY()
public:
    // BlueprintNativeEvent: C++ default + optional BP override
    UFUNCTION(BlueprintNativeEvent, BlueprintCallable, Category="Interaction")
    FText GetInteractPrompt() const;

    // BlueprintImplementableEvent: BP-only implementation
    UFUNCTION(BlueprintImplementableEvent, BlueprintCallable, Category="Interaction")
    void OnInteracted(AActor* Interactor);
};
```

The `U`-prefixed class needs no body beyond `GENERATED_BODY()`. The `I`-prefixed class is what
C++ code inherits and what holds function declarations.

## UINTERFACE specifiers (`ObjectMacros.h`:925–940)

| Specifier | Effect |
|---|---|
| `Blueprintable` | Allows Blueprint classes to implement the interface |
| `NotBlueprintable` | Prevents Blueprint from implementing; default unless `Blueprintable` |
| `MinimalAPI` | Exports only type info for Cast<>/vtable; keeps compile overhead low |

A `Blueprintable` interface may contain only `BlueprintNativeEvent` and
`BlueprintImplementableEvent` functions (plus C++ virtual functions). Adding non-event
BlueprintCallable C++ functions to a `Blueprintable` interface requires marking them with
`meta=(CannotImplementInterfaceInBlueprint)` on the function, or using
`NotBlueprintable`/`meta=(CannotImplementInterfaceInBlueprint)` on the interface.

## The three function types on an interface

### 1. Pure C++ virtual (not visible to Blueprint)

```cpp
virtual bool CanInteractWith(const AActor* Initiator) const = 0;
```

Can provide a default body in the `.cpp`. Blueprint classes cannot override it — use only when the
interface is C++-only.

### 2. BlueprintImplementableEvent

No C++ body. Blueprint must implement it; C++ cannot. Calling from C++ dispatches to the BP
implementation or does nothing (void) / returns zero (non-void) if unimplemented.

### 3. BlueprintNativeEvent

Provide `_Implementation` as the C++ fallback. Blueprint may override it.

```cpp
// Declaration in I-class (header):
UFUNCTION(BlueprintNativeEvent, BlueprintCallable, Category="AI")
float ScoreThreat(const AActor* Target) const;

// C++ default (header or cpp):
float ScoreThreat_Implementation(const AActor* Target) const;
```

## Implementing an interface in C++

```cpp
// MyTrap.h
#include "MyInteractable.h"

UCLASS(Blueprintable)
class ATrap : public AActor, public IMyInteractable
{
    GENERATED_BODY()
public:
    virtual FText GetInteractPrompt_Implementation() const override;
    // OnInteracted has no _Implementation — it is BlueprintImplementableEvent
};
```

Note: `BlueprintImplementableEvent` functions must **not** have a C++ `_Implementation`; they
have no C++ body at all.

## Implementing an interface in Blueprint

In Class Settings → Interfaces, add the interface. Blueprint then shows event/function nodes to
implement. When a Blueprint implements the interface, the `U`-prefixed class is **not** added to
the C++ vtable — `Cast<IMyInteractable>(SomeBP)` returns `null`. Use `Execute_` instead.

## Calling interface functions from C++

There are three call sites, each with different behavior:

| Call site | Routes to BP? | Use when |
|---|---|---|
| `IMyInteractable::Execute_GetInteractPrompt(Obj)` | yes | Always use this for `BlueprintNativeEvent`/`BlueprintImplementableEvent` — the only safe universal call |
| `Cast<IMyInteractable>(Obj)->GetInteractPrompt_Implementation()` | no | Intentionally want only the C++ default, skipping any BP override |
| Direct vtable call through raw `IMyInteractable*` | no | C++-only interfaces where BP never implements |

`Execute_<FuncName>` is generated by UHT in the `.generated.h` file. It inspects the object at
runtime and routes to the Blueprint override when present, or the `_Implementation` otherwise.

```cpp
// Correct universal dispatch:
if (SomeActor->Implements<UMyInteractable>())
{
    FText Prompt = IMyInteractable::Execute_GetInteractPrompt(SomeActor);
}

// Check and cast (C++ only):
IMyInteractable* Ptr = Cast<IMyInteractable>(SomeActor);
if (Ptr)  // null for BP-implemented interfaces
    Ptr->GetInteractPrompt_Implementation();
```

## Checking interface implementation

Three ways; behavior differs when the implementation is in Blueprint:

```cpp
// Works for both C++ and Blueprint implementations:
bool bHas = SomeActor->Implements<UMyInteractable>();
bool bHas2 = SomeActor->GetClass()->ImplementsInterface(UMyInteractable::StaticClass());

// Null if implemented only in Blueprint (vtable check):
IMyInteractable* Ptr = Cast<IMyInteractable>(SomeActor);
```

Prefer `Implements<>` for the check when you need to handle both C++ and BP implementors.

## TScriptInterface — safe storage of interface + object

When you want to store a reference to both the `UObject` and its interface pointer together:

```cpp
TScriptInterface<IMyInteractable> StoredInteractable;

if (SomeActor->Implements<UMyInteractable>())
    StoredInteractable = TScriptInterface<IMyInteractable>(SomeActor);

// Retrieve later:
UObject* Obj = StoredInteractable.GetObject();       // the UObject
IMyInteractable* I = StoredInteractable.GetInterface();  // the I* (null for BP implementors)
```

`TScriptInterface` holds a strong reference to the `UObject`, preventing GC. Use it in
`UPROPERTY` members where you need to keep both pointers alive.

## Common mistakes

- **Calling `Execute_` on a non-interface function** — `Execute_` is only generated for
  `BlueprintNativeEvent`/`BlueprintImplementableEvent` functions inside an interface.
- **Providing a body for `BlueprintImplementableEvent`** — compile error; UHT owns the body.
- **Forgetting `GENERATED_BODY()` in the `I`-class** — UHT won't generate dispatchers.
- **Mixing `BlueprintCallable` without `BlueprintNativeEvent`/`BlueprintImplementableEvent`** —
  a plain `BlueprintCallable` virtual on a `Blueprintable` interface causes a UHT error unless
  marked `meta=(CannotImplementInterfaceInBlueprint)`.
- **Casting instead of using `Execute_`** — works for C++ implementors, silently skips Blueprint
  implementors.
- **Using `TSubclassOf<UMyInteractable>`** — interfaces are not classes; use
  `TScriptInterface<IMyInteractable>` for typed interface storage.

## Version notes

- The `Implements<>` template and the `Execute_` dispatch pattern are stable across UE 5.x.
- `TScriptInterface` has been available since UE4; prefer it over raw `I*` pointers in
  UPROPERTY members.
